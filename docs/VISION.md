# SmartFill-Scholar — 核心願景

> 最後更新：2026-03-09

## 一句話定位

**SmartFill-Scholar 是一個 Personal Knowledge API** —— 它將個人或組織的散落資料（資料庫 + 文件 + 知識庫）統一為可程式化查詢的知識層，表單填寫只是其中第一個應用。

---

## 為什麼做這個？

學術工作者每年花大量時間在重複性文書工作上：

- 科技部計畫書：每年重寫，80% 內容跟去年類似
- 學校行政表單：個人資料填了又填，每次格式不同
- 升等/評鑑資料：從十幾份文件中拼湊同樣的內容

**根本問題不是「表單太多」，而是「知識散落在各處」。**

個人資料在 HR 系統、論文在 Google Scholar、計畫書在硬碟資料夾、經歷在 CV 裡。每次需要填表，都在做人肉 ETL（Extract-Transform-Load）。

---

## 核心洞察

```
┌─────────────────────────────────────────────────────┐
│                你以為你在做的事                        │
│         「幫使用者自動填表單」                          │
├─────────────────────────────────────────────────────┤
│                你真正在做的事                          │
│    「建立一個可查詢的個人知識 API」                      │
│                                                      │
│     Input:  散落的資料 (SQL + 文件 + 向量庫)           │
│     Engine: 意圖理解 + 智能路由 + RAG 生成             │
│     Output: 結構化的答案（表單、文字、報告...）          │
└─────────────────────────────────────────────────────┘
```

表單填寫 = 第一個 output adapter。
但同一套引擎可以回答「我去年的研究成果摘要是什麼？」「幫我起草給科技部的信」「把我的 CV 更新到最新」。

## 架構哲學：三層模型

```
┌─────────────────────────────────────────────────┐
│          Layer 3: 應用層 (Applications)           │
│                                                   │
│   📝 表單填寫    💬 知識問答    ✉️ 郵件草稿        │
│   📊 報告生成    📋 合規檢查    🔍 文獻回顧        │
│                                                   │
│   每個「應用」= 一組 prompt + output formatter     │
├─────────────────────────────────────────────────┤
│          Layer 2: 智能路由層 (Router)             │
│                                                   │
│   Intent Router: 「這個問題該去哪裡找答案？」       │
│                                                   │
│   ┌──────────┐  ┌──────────┐  ┌──────────┐      │
│   │ SQL 查詢  │  │ 向量搜尋  │  │  跳過    │      │
│   │ (精確)   │  │ (語意)   │  │ (人工)   │      │
│   └──────────┘  └──────────┘  └──────────┘      │
├─────────────────────────────────────────────────┤
│          Layer 1: 資料擷取層 (Ingestion)          │
│                                                   │
│   📂 資料夾監控   📄 文件解析   🔄 增量更新         │
│   🏷️ 自動標籤    📊 Metadata   ⏰ 排程索引         │
│                                                   │
│   結構化 (SQLite)  │  非結構化 (ChromaDB)          │
│   who / what       │  context / why / how          │
└─────────────────────────────────────────────────┘
```

### 設計原則

1. **Layer 1 是基礎** —— 資料不進來，上面什麼都做不了
2. **Layer 2 是核心競爭力** —— 「知道去哪找」比「能找到」更重要
3. **Layer 3 是可替換的** —— 每個應用都是薄薄一層 adapter

## 誰會用？

### 短期：學術研究者

| 使用者 | 痛點 | 系統提供的價值 |
|--------|------|---------------|
| 大學教授 | 每年填相同的計畫書/評鑑表 | 從知識庫自動填寫，一鍵產出 |
| 研究生 | 文獻回顧耗時、不知道老師過去做了什麼 | 語意搜尋 + 自動摘要 |
| 系所助理 | 重複整理教師資料 | 結構化查詢 + 批量匯出 |

### 中期：泛化到任何需要「知識管理」的領域

| 領域 | Entity | 知識庫內容 | 應用 |
|------|--------|-----------|------|
| **學術** | 教授/研究者 | 論文、計畫書、成果報告 | 表單填寫、文獻回顧 |
| **企業** | 公司/部門 | SOP、會議記錄、專案文件 | 員工問答、報告生成 |
| **法律** | 客戶/案件 | 判例、合約、法規 | 合約審查、法規查詢 |
| **醫療** | 醫師/科室 | 病歷摘要、臨床指引 | 病歷填寫、用藥查詢 |
| **個人** | 自己 | 筆記、日記、收藏文章 | 個人知識庫問答 |

## 應用場景展開

基於同一套 SQL + Vector + LLM Router 引擎，幾乎零成本就能延伸：

### Tier 1 — 已有基礎設施，只需加 adapter

| 應用 | 做法 | 一句話價值 |
|------|------|-----------|
| **表單自動填寫** | ✅ 已完成 | 上傳表單 → AI 填寫 → 人工審查 |
| **知識問答 (Chat)** | ✅ 已完成 | 「王教授的研究專長是什麼？」 |
| **郵件草稿生成** | ✅ 已完成 | 給定情境 + 收件人，自動擬稿 |
| **報告生成** | ✅ 已完成 | 主題 + 類型 → 結構化研究報告 |
| **個人 FAQ Bot** | 前端加 chatbot UI | 系所網頁上的智能客服 |

### Tier 2 — 需要少量新元件

| 應用 | 需要新增 | 一句話價值 |
|------|---------|-----------|
| **計畫書段落生成** | Section template engine | 從舊計畫書 + 論文中生成新段落 |
| **自動文獻回顧** | Citation formatter + structure | 給定主題，產出 structured review |
| **會議準備包** | Calendar integration + summarizer | 開會前自動彙整相關文件摘要 |

### Tier 3 — 需要架構擴展

| 應用 | 需要新增 | 一句話價值 |
|------|---------|-----------|
| **跨人員知識圖譜** | Entity 關聯 + Graph DB | 「誰跟誰合作過？哪些計畫有交集？」 |
| **合規檢查** | Rule engine + checklist | 計畫書格式 / 必填欄位自動驗證 |
| **版本差異追蹤** | Document diff + timeline | 「這份跟上一版改了什麼？」 |

## 跟現有產品的差異

| 維度 | Google NotebookLM | Notion AI | SmartFill-Scholar |
|------|-------------------|-----------|-------------------|
| 資料來源 | 上傳文件 | Notion 頁面 | SQL + 向量 + 文件系統（混合） |
| 結構化資料 | ❌ | 有限 | ✅ 完整 SQL 查詢 |
| 路由決策 | ❌ | ❌ | ✅ LLM 自動決定資料來源 |
| 表單填寫 | ❌ | ❌ | ✅ 核心功能 |
| 自動索引 | ❌ | 頁面內 | ✅ 檔案系統監控（規劃中） |
| 自部署 | ❌ | ❌ | ✅ 完全本地 |
| 隱私 | 雲端 | 雲端 | ✅ 資料不離開本機 |

**核心差異：混合檢索 + 智能路由。** 不只是 RAG，而是「先判斷去哪找，再用對應方式找」。

**終極目標**：每個人（或每個組織）都有一個私有的、可查詢的知識 API，任何需要「從我的資料中找答案」的場景都走這一層。

---

## 指導原則

1. **資料先行**：沒有好的 Ingestion，就沒有好的 Output
2. **路由是智慧**：系統的價值在於「知道去哪找」，而不只是「能找到」
3. **Output 是可替換的**：不要把太多邏輯綁在某一種輸出格式上
4. **隱私第一**：所有資料本地化，不依賴雲端儲存
5. **漸進式複雜度**：先做好一個場景（學術表單），再泛化
